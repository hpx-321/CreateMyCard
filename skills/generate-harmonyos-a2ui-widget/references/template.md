# 卡片尺寸模板与布局规约

> **用途：** 确定需求可行后（见 [capability.md](capability.md)），根据目标卡片尺寸选择模板和布局策略。本文件只负责尺寸、布局、配色和背景图策略；协议和字段细节分别见 [protocol-core.md](protocol-core.md) 与 [component-field-reference.md](component-field-reference.md)。

## 鸿蒙桌面卡片网格系统

鸿蒙桌面卡片使用网格单元布局，1 单元 ≈ 80-100vp（因设备 DPI 而异）。以下使用近似 vp 值，布局应以百分比/弹性尺寸为主，不硬编码宽度。

| 规格 | 列×行 | 近似尺寸 (vp) | 宽高比 | 适用场景 |
|------|-------|-------------|--------|---------|
| **1×2** | 1×2 | ~80×160 | ~1:2 | 微卡：单指标、图标+数值 |
| **2×2** | 2×2 | ~160×160 | ~1:1 | 方卡：主指标+上下文、状态+操作 |
| **2×4** | 2×4 | ~160×320 | ~1:2 | 中卡：列表、纵向详情、多条明细 |
| **4×4** | 4×4 | ~320×320 | ~1:1 | 大卡：仪表盘、组合信息 |

## 泛化建模流程

不要按业务名选模板。先把用户需求抽象为信息角色，再映射到尺寸和组件。

| 角色 | 问题 | 典型组件 | 限制 |
|------|------|------|------|
| `primary` | 用户一眼最需要知道什么？ | 大号 `Text`、小图标、`Progress` | 每卡只允许 1 个主焦点 |
| `context` | 主信息的时间、地点、来源、原因是什么？ | 小号 `Text`、半透明容器 | 1×2 可省略 |
| `details` | 有哪些辅助槽位或同类明细？ | `Row`/`Column`/`List`/`Grid` | 超过 3 条至少 2×4 |
| `action` | 用户能做什么轻操作？ | `Button` 或可点击 `Row` + `Text` | 桌面卡片避免复杂表单 |
| `state` | 是否需要加载、空态、异常或断点分支？ | `If` | 分支都必须有引用组件 |

布局选择顺序：
1. 找唯一 `primary`。
2. 统计 `context`、`details`、`action` 数量。
3. 用最小可承载尺寸，不为留白升级。
4. 将角色放入布局槽位，而不是复用示例业务文案。
5. 若角色太多，先裁剪/合并辅助信息，再升级尺寸。

## 通用布局铁律（所有尺寸）

以下规则适用于所有卡片尺寸，违反将导致渲染异常或布局错乱：

### 根容器
- `root` 组件 `styles.width` 必须为 `"100%"`（填满卡片容器）。
- `root` 组件 `styles.height` 不设固定值，由内容撑开。仅在已知精确卡片像素尺寸时设置 `constraintSize.maxHeight`。
- 根容器优先使用 `Column`，除非 4×4 大卡且需求明确为横向布局。

### 间距规范
- 所有相邻区域使用 `itemMargin` 或 `space` 明确设置间距。不得依赖默认值。
- 卡片外层 `styles.padding` 与内容区间距遵循下表：

| 卡片规格 | padding | itemMargin | 说明 |
|---------|---------|-----------|------|
| 1×2 | 8-10 | 4-6 | 极致紧凑 |
| 2×2 | 12-14 | 8-10 | 标准紧凑 |
| 2×4 | 14-16 | 10-12 | 适度留白 |
| 4×4 | 16-20 | 12-16 | 充分呼吸感 |

### 字体规范
- `fontSize` 必须为**纯数字**（默认 vp），不得写成 `"16fp"` 等字符串。
- `maxLines` + `textOverflow: "ellipsis"` 必须成对出现在所有可能超长的文本上。

| 卡片规格 | 主标题 | 正文 | 大数字 | 辅助文字 |
|---------|--------|------|--------|---------|
| 1×2 | 13-14 | 11-12 | 28-32 | 9-10 |
| 2×2 | 15-17 | 13-14 | 32-42 | 10-11 |
| 2×4 | 16-18 | 13-15 | 36-48 | 11-12 |
| 4×4 | 18-22 | 14-16 | 42-56 | 12-13 |

### 颜色规范
- 使用 `#RRGGBB` 或 `#AARRGGBB` 格式。
- 文本色阶（深色背景上使用白色透明阶）：
  - 主要文字：`#E6000000`（浅底）/ `#F2FFFFFF`（深底）
  - 次要文字：`#CC000000` / `#E6FFFFFF`
  - 辅助文字：`#99000000` / `#B3FFFFFF`
  - 最弱文字：`#66000000` / `#73FFFFFF`
- 强调色使用 `theme.primaryColor` 或语义色。

### 对比度与组件边界规则

可读性优先于装饰性。生成时必须保证文字、背景、子卡片和按钮之间有清晰层级，避免同色系低对比导致内容糊在一起。

1. **文字层、区域层、背景层必须分离：** 文字颜色要与其承载区域可读，区域颜色要与外层背景可辨，背景本身不能抢占正文与操作区。生成时不要只看某一层单独成立，要看三层叠加后的区分度。
2. **文字与承载底色分离：** 深色/渐变背景上主要文字使用 `#FFFFFFFF` 或 `#F2FFFFFF`，辅助文字不低于 `#B3FFFFFF`；浅色背景上主要文字使用 `#E6000000`，辅助文字不低于 `#99000000`。不要在任何背景上使用与底色同色相、近明度的文字。
3. **区域先看局部背景，不看整卡平均色：** 按钮、信息卡、状态块必须与它实际落点的背景比较，尤其是底部或右下收束区。若局部背景已偏深、偏亮或饱和度较高，区域色不得继续沿用同色相中近明度变体。
4. **CTA 必须同时通过内对比和外对比：** 内对比指“按钮文字 vs 按钮底色”，外对比指“按钮轮廓 vs 按钮所在局部背景”。两者缺一不可；只看文字能读、不看按钮是否融进背景，视为不合格。
5. **按钮必须比背景更醒目：** CTA 的文字色与按钮底色必须形成明显明暗差。深色局部背景优先使用近白字 + 清晰描边的半透明浅色 pill，或高对比实底；浅色按钮应使用深色文字；禁止任何“同色相 + 近明度 + 弱边界”的低对比组合。`亮蓝底 + 深蓝尾色` 只是其中一个例子。
6. **子卡片边界双保险：** 内层信息卡至少使用两种边界手段中的一种，复杂或渐变背景上建议同时使用：背景明度差（如 `#1EFFFFFF`-`#2EFFFFFF` 半透白）+ 1vp 浅描边（如 `#33FFFFFF`）或适度阴影。
4. **模块之间留白可感知：** 相邻模块的间距应比模块内部行距更大。2×2 中模块间建议 8-10vp，模块内部文本间距建议 2-4vp。
7. **同级模块要同规格：** 多个同级子卡片的 `borderRadius`、`padding`、背景透明度、描边强度保持一致；主操作按钮可更高对比，但不能和普通信息卡混成同一视觉等级。
8. **分隔线谨慎使用：** 浅色卡片可用 `borderWidth:1` 或细分隔线表达模块边界；深色渐变卡片优先用半透明子卡片和描边，避免硬分割线破坏整体感。

### 通用视觉组织规则

桌面卡片不是普通页面缩略版。生成时先确定信息层级，再选择布局骨架，最后才填充组件。

1. **一卡一主旨：** 每张卡片只能有 1 个主任务或主指标。其他内容只能作为解释、状态、趋势或操作入口出现。
2. **三层以内：** 推荐结构为“主信息层 + 上下文层 + 操作层”。1×2 通常只保留 1-2 层，2×2/2×4 可使用 2-3 层，4×4 才允许更多模块组合。
3. **视觉焦点唯一：** 大数字、大图标、主按钮三者最多同时突出 1 个；其余元素通过字号、透明度、位置或半透明容器降级。
4. **边缘留白稳定：** 外层 padding 控制安全边距，内部 `itemMargin` 控制节奏；不要用空 Text、空 Row 或过大的 padding 人为撑开布局。
5. **容器承载语义：** 半透明子卡片用于承载结构化辅助信息，pill 按钮用于承载主要操作，纯文本只用于标签和短状态。
6. **图标少而准：** 图标/emoji 是视觉锚点，不是每行文字的前缀。1×2 最多 1 个，2×2 最多 1-2 个，2×4/4×4 可按模块增加但必须服务于分组识别。

### 渐变背景设计方法

渐变背景应服务于可读性和信息层级，而不是单纯堆叠品牌色。所有尺寸可按下列模型生成，2×2 默认强制使用。

1. **受光色（stop 0）：** 放在左上或信息起点区域。应比主题色更亮、更低攻击性，可使用浅蓝、雾绿、暖灰、奶油橙等语义化提亮色。
2. **主题色（stop 0.45-0.55）：** 承接业务语义和品牌识别，是卡片的主情绪。建议使用低到中饱和色，避免同时使用多个高饱和色。
3. **收束色（stop 1）：** 放在右下或视觉终点区域。比主题色更深，用于压住边角和提高白字对比，但不要直接压到纯黑。
4. **同源派生：** 内层子卡、描边、辅助文字、CTA 背景应从同一渐变色系派生。深色底常用 `#18FFFFFF`-`#2EFFFFFF` 半透白子卡，描边使用 `#24FFFFFF`-`#40FFFFFF`。
5. **方向选择：** 小卡优先 `"RightBottom"`，让视线从主信息流向辅助信息或操作入口；横向大卡可根据主内容方向选择，但避免纯水平/垂直导致色块割裂。

**通用渐变质量检查：**
- 首色和尾色不要同时高饱和。
- 尾色不要近黑，除非夜间、睡眠、安全等语义确实需要深色氛围。
- 辅助文字必须与底色保持对比，深底优先用白色透明阶。
- CTA 不应与背景脱节；默认使用同色系半透明 pill，强确认操作才使用高对比实底。
- 任一区域若落在高饱和、低亮度或高亮度的局部背景上，区域色和文字色都不得继续沿用同色相近明度变体；优先切换到更大的明度差、更清晰的描边，或增加独立承载底。
- 渐变中若存在大面积中亮度区域，文字和按钮不要继续使用同亮度同色相；改用白字、深字或增加半透明承载底。
- 内层子卡片不能只靠颜色相近的透明底表达边界；至少增加描边、阴影、明度差或更大的模块间距。

### 组件数量硬限

桌面卡片以 3 秒扫读为目标。超过以下上限的组件树应裁剪：

| 卡片规格 | 最多组件数 | 最多可见文本行 | 最多按钮 |
|---------|-----------|-------------|---------|
| 1×2 | 8 | 3 | 1 |
| 2×2 | 15（图像叠层/设备状态类最多 22） | 6 | 2 |
| 2×4 | 22 | 12 | 3 |
| 4×4 | 35 | 20 | 4 |

## 尺寸专属布局规约

### 1×2 微卡（~80×160vp）

**核心原则：极致精简，单列纵向，一瞥即得。**

```
布局铁律：
├── 根组件：Column，单列，禁止 Row/Grid/List
├── 最多 2 个信息区 + 可选 1 个按钮
├── 图标不超过 28×28vp
├── 不使用 Image（空间不足以展示有意义的图片）
├── 禁止使用 Progress（条形占用纵向空间）
├── 禁止使用 List/Grid（空间不足）
└── 文本全部 maxLines:1
```

**推荐模板：**

```
模板 A — 图标+指标
┌──────────┐
│  ☀️      │  ← iconEmoji (Text, fontSize:24)
│  28°     │  ← primaryValue (Text, fontSize:32, fontWeight:bold)
│  上海     │  ← secondaryLabel (Text, fontSize:11)
└──────────┘

模板 B — 标题+数值+操作
┌──────────┐
│ 标题     │  ← title (Text, fontSize:14, fontWeight:bold)
│ 副标题   │  ← subtitle (Text, fontSize:11)
│ [⚡ 操作]│  ← actionBtn (Button, fontSize:13)
└──────────┘

模板 C — 单指标+进度条
┌──────────┐
│ 指标名称  │  ← label (Text, fontSize:12)
│ 8,420    │  ← value (Text, fontSize:28, fontWeight:bold)
│ ●●●●●○○  │  ← progress (Progress, type:capsule, height:3)
└──────────┘
```

**1×2 反模式（禁止）：**
- ❌ 使用 Row 放多个并排信息区
- ❌ 大号 Progress ring/eclipse
- ❌ 多行 Text 不设 maxLines
- ❌ 使用 List 组件
- ❌ padding 超过 12vp
- ❌ fontSize 超过 36

### 2×2 方卡（~160×160vp）

**核心原则：先按信息结构选布局，再用渐变、图片或内层容器增强层次。**

#### 2×2 布局选择规则

| 信息结构 | 首选布局 | 判断标准 |
|---------|---------|---------|
| 1 个主指标 + 2-3 个辅助槽位 | 象限网格 | 信息可自然拆成主指标、视觉锚点、详情、操作四类 |
| 1 个主状态/结论 + 1 个上下文 + 1 个操作 | 主任务分层卡 | 用户最可能先看状态，再决定是否点击操作 |
| 1 个主状态/指标 + 1 个详情卡 + 1 个圆形操作 | 上下分区行动卡 | 顶部适合大数值/天气/状态，底部适合一张小卡和一个触达入口 |
| 真实产品/设备/场景图片 + 2-3 个状态槽位 | 图像背景叠层卡 | 图片是第一视觉，文字和数据需要避开主体并放在半透明承载区 |
| 2-3 条同类列表项 | 单列流式 | 内容同质，不需要四象限分权 |
| 多个同级开关/设备/统计块 | 升级 2×4 或 4×4 | 2×2 无法同时保证可读性和触达面积 |

生成时优先判断“用户扫一眼最需要知道什么”，再决定布局。不要因为示例中出现过某种业务就固定套用某个模板。

#### 新模板抽象：2×2 稳定骨架

当前 2×2 模板参考位于 [dsl-templates/](dsl-templates/)：`top-left-weather-schedule.dsl.jsonl`、`parents-care-weather.dsl.jsonl`、`free-clip-2-widget/free-clip-2.dsl.jsonl`。它们的共性不是业务，而是“固定 160vp 画布 + 单一主焦点 + 半透明承载区 + 清晰触达入口”。

**骨架 A：分层主任务卡**

适合天气+日程、会议、提醒、状态处理等“主指标/主结论 + 上下文 + 主操作”场景。

```
root (Column, height:160, padding:10-12, itemMargin:6-8)
├── heroRow (Row, 图标 token + 右侧主指标)
├── contextCard (Row/Column, 半透明信息承载区)
└── actionPill (Row, 底部整宽或近整宽操作)
```

生成要点：
1. 顶部只保留一个强焦点：大数字 34-38vp、主状态或图标 token，另一个顶部元素降级为辅助。
2. `contextCard` 使用 `#22FFFFFF`-`#28FFFFFF` 半透明底、1vp 白色透明描边、14-16vp 圆角；可用细 `Divider` 做左侧强调线，但不要把每行都加图标。
3. CTA 使用可点击 `Row` + `Text` 更稳定，默认 32-38vp 高、半透明浅底、白字、清晰描边；如果是拨打/确认等强动作，可用高对比实底并加轻阴影。

**骨架 B：上下分区行动卡**

适合关怀、天气、告警、家庭服务、设备状态等“顶部状态 + 底部详情/动作”场景。

```
root (Column, height:160, padding:12, itemMargin:10, justifyContent:spaceBetween)
├── topRow (Row, 左侧大指标 + 右侧视觉状态)
└── bottomRow (Row, 左侧固定信息卡 + 右侧圆形操作)
```

生成要点：
1. 左上大指标推荐 36-40vp，右上图标推荐 36-40vp；二者不同时使用高饱和颜色抢焦点。
2. 底部信息卡使用固定尺寸，常见范围 72-80 × 54-60vp；圆形按钮使用 54-60vp，`borderRadius` 为半宽。
3. 顶部信息区可裸露在渐变上，底部详情和操作必须有独立承载底、描边或阴影。

**骨架 C：图像背景叠层卡**

适合产品、设备、场馆、人物或有真实主视觉的卡片。图片必须服务识别，不能只是装饰背景。

```
root (Stack, height:160, borderRadius:20-24, clip:true)
├── productImage / sceneImage (Image, width/height:100%, objectFit:cover)
└── contentLayer (Column, width/height:100%, padding:12-16)
```

生成要点：
1. 背景图用 `Image` 作为 Stack 第一层，不把关键图片塞进 `backgroundImage`；`src` 可绑定资源路径，`objectFit:"cover"`，根容器必须 `clip:true`。
2. 文案层要避开图片主体：用固定宽度文本块、底部半透明状态块或 `justifyContent:"spaceBetween"` 分配空间。确需占位时，只能在固定 160vp 画布内使用显式高度的透明 spacer。
3. 浅色图片上使用深色文字（如 `#FF111827`）和 `#80FFFFFF` 半透明白卡；深色图片上使用白字和 `#26000000`-`#40000000` 暗色遮罩或半透明深卡。
4. 图像叠层的组件数可到 22，但必须控制可见文本不超过 6 行，且只保留 1 个主要可点击入口。

**2×2 共性参数：**

| 项目 | 建议 |
|------|------|
| 画布 | 明确 2×2 渲染时可设 `height:160`；根容器 `width:"100%"`、`borderRadius:20-24`、`clip:true` |
| 外边距 | `padding:10-14`；图像叠层顶部可 14-16、底部 6-10 |
| 字号 | 主指标 34-40，标题 14-17，辅助 9-12，操作文字 13-14 |
| 内层容器 | 半透明白 `#22FFFFFF`-`#80FFFFFF`，描边 `#33FFFFFF`-`#66FFFFFF`，圆角 14-18 |
| 背景 | 信息卡优先语义渐变；产品/设备/真实场景优先图片叠层；不要同时让复杂图片和高饱和渐变争抢焦点 |

#### 备选布局：象限网格（多槽位信息）

2×2 最高效的布局模式是**上下两行等分 + 每行左右两列**的四象限结构。此模式经过端侧渲染验证，能可靠避免排版错位。

```
结构骨架（强制）：
root (Column, itemMargin:0, matchParent/matchParent)
├── top_row (Row, layoutWeight:1)
│   ├── top_left (Column, layoutWeight:1, height:matchParent)
│   └── top_right (Column, layoutWeight:1, height:matchParent)
└── bottom_row (Row, layoutWeight:1)
    ├── bottom_left (Column, layoutWeight:1, height:matchParent)
    └── bottom_right (Column, layoutWeight:1, height:matchParent)
```

**象限对齐约定（四象限角色模型）：**

| 象限 | 内容角色 | justifyContent | alignItems | 典型填充 |
|------|---------|---------------|------------|---------|
| 左上 | **主指标区** — 卡片最重要的数值 | `start` | `start` | 大字指标 + 副标签 |
| 右上 | **视觉锚点区** — 吸引目光的图形元素 | `start` | `end` | 图标/emoji + 摘要文字 |
| 左下 | **详情卡片区** — 辅助信息的结构化展示 | `end` | `start` | 固定尺寸子卡片（指标卡/状态卡） |
| 右下 | **操作入口区** — 用户可触达的交互 | `end` | `end` | 圆形按钮 / 小型操作区 |

四象限角色与具体业务解耦：无论卡片是天气、健康、设备还是日程场景，四个象限承担的角色不变。生成时根据用户 query 将业务内容填入对应角色槽位。

```
通用象限布局（角色视图）：
┌──────────────────────┐
│ 左上 (start/start)   │  右上 (start/end)
│ 主指标                │        🎯 视觉锚点
│ 辅助标签              │       摘要信息
│                      │
├──────────────────────┤
│ 左下 (end/start)     │  右下 (end/end)
│ ┌──────────┐         │
│ │ 详情卡片  │         │        ⚡ 操作入口
│ │ 辅助信息  │         │
│ └──────────┘         │
└──────────────────────┘
```

**象限式 2×2 生成要点（强制执行）：**

1. **等分行高：** 上下两个 Row 必须使用相同的 `"layoutWeight":1`。不得省略任一侧，否则行高不均。
2. **等分列宽：** 每个 Row 内的左右 Column 使用相同的 `"layoutWeight":1` 和 `"height":"matchParent"`。
3. **root 间距为 0：** `"itemMargin":0`，行间不留空隙。象限内的留白通过 Column 的 `padding` 控制。
4. **固定子卡片：** 象限内的小卡片（如辅助指标卡）必须使用固定 `width`/`height` 数值（vp），不得使用百分比或 `matchParent`，防止尺寸漂移。
5. **所有文本 `flexShrink:0`：** 防止弹性收缩导致字号不一致。
6. **列 padding 对称：** 左右 Column 的 `padding.top` / `padding.bottom` 保持相同值，确保象限基线对齐。

**渐变色彩规范（2×2 强制执行）：**

2×2 方卡面积有限，纯色背景容易显得扁平单调。使用 2-3 色对角渐变可显著增强视觉层次。

**核心原则：渐变色彩必须根据用户 query 语义动态选择，禁止使用固定色板。**

**语义色板速查表：**

| 语义 | 触发信号 | 建议色板（3 色） | 辅助文字色 |
|---------|-----------|-----------------|---------------------|
| 清爽/环境 | 自然、天气、空气、户外、位置状态 | 浅天蓝→低饱和蓝→深海蓝 | `#FFD8E8FF` |
| 温暖/关怀 | 家庭、陪伴、提醒、服务、生活照护 | 柔暖橙→珊瑚粉→柔和紫 | `#FFE8D8F0` |
| 活力/健康 | 运动、健康、能量、完成度、成长 | 薄荷绿→青绿→深蓝绿 | `#FFD8F0E8` |
| 专注/效率 | 任务、会议、项目、排班、时间压力 | 低饱和蓝→紫罗兰→靛蓝 | `#FFD8E0F8` |
| 控制/设备 | 设备、安全、开关、传感器、系统状态 | 深灰蓝→暗紫→墨色 | `#FFD0D0E0` |
| 财务/交易 | 金额、收益、账单、价格、库存 | 淡青绿→青蓝→深蓝绿 | `#FFD8F0EC` |
| 警示/异常 | 风险、低电、延误、告警、失败 | 柔黄→琥珀→深赭红 | `#FFFFE8CC` |
| 默认兜底 | 无法归类 | 低饱和品牌主色→邻近深色→深中性色 | `#FFE8E8F0` |

**色板设计原则：**

1. **首色**（色标 0）：明亮、饱和度适中，占据卡片左上主要区域
2. **中间色**（色标 0.45-0.55）：首色的邻近色或同色系深变体，过渡自然，视觉重心落在卡片中央
3. **尾色**（色标 1.0）：深色收束，提供视觉锚点和文字对比度
4. **文字色**：深色渐变底上主要文字使用 `#FFFFFFFF`（纯白）；辅助文字向首色偏移（见上表辅助文字色列）

**高质量卡片的氛围渐变约束：**

1. **左上受光区：** 首色不要直接使用高饱和品牌主色，优先使用更亮、更柔和的同类色或邻近色，承接小图标、小标签和留白。
2. **中部主题区：** 中间色负责卡片主情绪，优先选择主题色的低饱和版本；避免多个高饱和色同时竞争视觉焦点。
3. **右下收束区：** 尾色要更深，但仍保留明确色相；避免直接压到近黑色，确保 2×2 卡片仍有轻盈感和呼吸感。
4. **内外层同源：** 内层信息卡、CTA、辅助字色都应从同一渐变色系派生，避免背景、按钮、文字各自使用互不相关的色彩。

**生成流程：**

```
1. 从用户 query 提取业务语义、情绪和状态强度
2. 匹配"语义色板速查表"选择色相方向
3. 按三段模型自行生成具体色值，不复制示例色值
4. 如无匹配 → 使用"默认兜底"色板
5. 辅助文字色跟随选定色板的推荐值，并检查局部背景对比
```

**完整示例（冷暖平衡渐变）：**

```json
"linearGradient": {
  "direction": "RightBottom",
  "colors": [
    ["#DED2A8", 0],
    ["#99A5F0", 0.48],
    ["#5F69C6", 1]
  ]
}
```

上述示例体现“受光色 + 低饱和主题色 + 深色收束”的通用结构。生成时必须替换为符合当前业务语义的色相；辅助文字可使用 `#FFF2ECFF`，主要文字使用 `#FFFFFFFF`。

**渐变字段约束：**

- 方向固定 `"RightBottom"`（对角线），不要在 2×2 中使用水平或垂直渐变。
- `colors` 为 `[颜色字符串, 色标位置]` 元组数组，色标直接嵌入颜色项。**禁止**使用独立的 `stops` 数组或 `repeating` 字段，渲染器不支持。
- 色标 2-3 个，中间色标建议在 0.45-0.55 区间（视觉重心在卡片中心）。
- 渐变写于 root 组件的 `styles` 内，与 `borderRadius` + `clip:true` 配合裁切。

**象限式子卡片规范：**

| 子卡片 | 推荐尺寸 (vp) | 背景 | 圆角 | 内部间距 |
|--------|-------------|------|------|---------|
| 辅助指标卡 | 72-78 × 52-58 | `#26FFFFFF` 半透白底 + `#33FFFFFF` 边框 1vp | 14-16 | itemMargin:3-4, padding:8 |
| 圆形操作按钮 | 44-48 × 44-48 | `#FFFFFFFF` 不透明白底 | 22-24（半宽=圆） | 仅 label 文字 |
| 小图标区 | 40-48 × 40-48 | 透明（继承背景） | 不需要 | 仅 emoji Text |

**象限式反模式（禁止）：**

- ❌ Row 不设 `layoutWeight:1` → 行高塌陷或溢出
- ❌ 左右 Column 不设 `height:"matchParent"` → 列高不对齐
- ❌ root 使用 `itemMargin > 0` → 行间出现不可控缝隙
- ❌ 子卡片使用 `width:"50%"` 或 `matchParent` → 宽度漂移、象限越界
- ❌ `fontSize` 超过 44 → 文字溢出 160vp 宽度
- ❌ 缺少 `flexShrink:0` → 文本被压缩变形
- ❌ 四象限用相同对齐 → 失去视觉层次
- ❌ 渐变方向用 `Top`/`Bottom` → 2×2 空间下单调，对角线更有层次
- ❌ 首尾色都高饱和、尾色接近纯黑 → 卡片质感发硬、信息层级变弱

#### 主任务分层卡：单列服务布局（状态 / 提醒 / 控制 / 行动类推荐）

当 2×2 卡片同时满足“**一个主状态/主数值 + 一个上下文信息 + 一个主要操作**”时，优先使用主任务分层卡，而不是把三段内容机械堆成等权模块。适用场景包括但不限于出行、会议、设备控制、健康状态、学习/专注、家庭提醒等。

```
推荐骨架：
root (Column, itemMargin:8-10)
├── heroRow (Row)
│   ├── heroLeft  (Column, 小图标/场景标签/辅助提示)
│   └── heroRight (Column, 主数值/主时间 + 地点或副标题)
├── contextCard (Column, 半透明浮层信息卡)
└── actionPill (Button, 底部整宽或近整宽主要操作)
```

**生成要点：**

1. **主次分层：** 顶区只保留 1 个绝对主视觉焦点（数值、状态、时间、进度或关键结论）；其余文字字号至少比主指标小 18%-24%。
2. **浮层信息卡：** 中层 `contextCard` 使用 `width:"100%"`，推荐 `backgroundColor:"#18FFFFFF" ~ "#2EFFFFFF"`、`borderWidth:1`、`borderColor:"#24FFFFFF" ~ "#40FFFFFF"`、`borderRadius:14-18`，建立“外卡 / 内卡”的层次。
3. **CTA 清晰可点：** 底部操作优先使用同色系半透明 pill，推荐 `height:34-38`、`borderRadius:18-20`、`borderWidth:1`、`borderColor:"#40FFFFFF" ~ "#66FFFFFF"`。半透明按钮默认使用白字；浅色实底按钮使用深字；禁止按钮文字与按钮底/卡片背景出现同色相近明度组合。必须同时检查“文字 vs 按钮底色”的内对比，以及“按钮轮廓 vs 底部局部背景”的外对比。
4. **图标去噪：** 图标应独立成小视觉 token（单独 Text / Image），不要让每一行文字都内嵌 emoji；整张卡最多保留 1-2 个图标锚点。
5. **留白与边缘：** heroRow、contextCard、actionPill 三层之间使用 8-10vp 节奏；底部按钮不要贴边，确保保留 10vp 以上的下沿呼吸感。

**主任务分层卡反模式（禁止）：**

- ❌ 头部状态、提醒、按钮三段全部做成等高等权块 → 没有视觉主次
- ❌ 中层提醒直接裸露在背景上，无半透明浮层或描边 → 信息黏在底图上
- ❌ 使用整块纯白大按钮 + 高饱和品牌色文字 → CTA 与背景断层
- ❌ 使用半透明按钮但仍配蓝/紫等同色文字 → 可点击区域和文字都不清晰
- ❌ 区域色、文字色与局部背景同色相近明度，导致按钮、信息卡或文字融进背景 → 属于高优先级设计错误
- ❌ 亮蓝按钮、亮蓝文字或亮蓝描边直接贴在深蓝/深紫底部收束区 → 上述错误的一种常见示例
- ❌ 每一行文字都带 emoji 前缀 → 视觉噪声高，接近聊天气泡而非系统卡片
- ❌ 渐变只追求“品牌色强对比”，忽略受光色、主题色和收束色的层次关系

**主任务分层卡参考实现：** 见 [dsl-templates/top-left-weather-schedule.dsl.jsonl](dsl-templates/top-left-weather-schedule.dsl.jsonl)。该示例仅展示结构和视觉层级，不应固定复用其中的业务文案或色值。

## 自适应策略（跨尺寸）

当用户未指定目标尺寸时，**不得默认生成 2×2**。必须先根据用户 query 推断内容承载量和排版结构，再选择最小可承载尺寸。当用户明确要求多尺寸适配时，使用 `If` 组件 + `$__WindowWidthBreakpoint` 实现：

### 未指定尺寸时的尺寸推断规则（强制）

生成前先从 query 中提取以下信号：

1. **信息槽位数**：主指标/主状态、辅助标签、列表项、统计项、提醒项、联系人/设备/日程条目等。
2. **操作数量**：按钮、开关、电话、导航、设置、查看详情等可点击入口。
3. **排版结构**：单指标、状态+操作、列表、双列对比、多卡片网格、仪表盘。
4. **可视组件估算**：Text、Button、Progress、Image、List/Grid 容器和子卡片的总量。

按“最小可承载尺寸”选择，不为留空而升尺寸，也不为了默认值压缩内容：

| 推断结果 | 推荐尺寸 | 典型结构 | 硬性判断 |
|---------|---------|---------|---------|
| 1 个主指标/状态 + 0-1 个辅助标签 + 0-1 个操作 | **1×2** | 单列微卡 | 组件数 ≤ 8、可见文本 ≤ 3 行、按钮 ≤ 1 |
| 1 个主指标/状态 + 2-3 个辅助槽位或 1 个上下文卡 + 1 个操作 | **2×2** | 象限网格或主任务分层卡 | 组件数 ≤ 15、可见文本 ≤ 6 行、按钮 ≤ 2 |
| 1 个主模块 + 3-5 条列表/明细，或 2-3 个操作，或需要展示短建议/预报/任务列表 | **2×4** | 纵向详情卡 | 组件数 ≤ 22、可见文本 ≤ 12 行、按钮 ≤ 3 |
| 多个同级模块、2×2 以上网格、仪表盘、设备矩阵、复杂统计或多分组列表 | **4×4** | 仪表盘/网格面板 | 组件数 ≤ 35、可见文本 ≤ 20 行、按钮 ≤ 4 |

**升级规则：**
- 估算组件数、文本行数或按钮数超过当前尺寸硬限时，升级到下一尺寸。
- 需要 `List` 且可见项超过 3 条时，至少使用 2×4。
- 需要 `Grid` 且同级项超过 4 个时，优先使用 4×4。
- 同时存在“列表 + 主操作 + 多个统计块”时，优先使用 2×4 或 4×4，不压成 2×2。

**降级规则：**
- query 只要求一个关键值、一个状态或一个快捷操作时，优先 1×2，不生成空洞的 2×2。
- 内容可以用一个主结论表达时，先裁剪辅助信息，再考虑升尺寸。
- 用户要求“简洁、小卡、快捷、只显示”时，在可读范围内优先选择更小尺寸。

### 策略 1：断点自适应（推荐）

利用 `$__WindowWidthBreakpoint` 判断卡片实际渲染宽度：

```json
{
  "id": "adaptiveRoot",
  "component": "If",
  "condition": "{{ $__WindowWidthBreakpoint == 'xs' || $__WindowWidthBreakpoint == 'sm' }}",
  "childrenIf": ["compactLayout"],
  "childrenElse": ["fullLayout"]
}
```

### 策略 2：多模板（按需选择）

一次性生成多套独立的卡片组件树，端侧根据卡片实际尺寸选择 Surface。不做在 DSL 内切换 Surface。

### 尺寸降级规则

当内容量超过卡片尺寸承载上限时，按以下优先级降级：
1. 裁剪 List 可见项数
2. 增大 `maxLines` 限制
3. 隐藏辅助信息（`visibility:"none"`）
4. 合并信息区
5. 最终手段：告知用户内容无法在此尺寸下展示，建议升级卡片尺寸

## 模板选择流程

```
用户需求 → 判断是否显式指定尺寸：
├── 已指定 → 匹配对应尺寸模板
└── 未指定 → 提取信息槽位/操作数量/排版结构/组件估算 → 选择最小可承载尺寸

选定尺寸后匹配模板：
├── 1×2：模板 A（图标+指标）/ B（标题+数值+操作）/ C（单指标+进度条）
├── 2×2：主任务分层卡 / 上下分区行动卡 / 图像背景叠层卡 / 象限网格 / 模板 E（列表+操作）/ F（多行状态）
├── 2×4：模板 G（大指标+详情+列表）/ H（进度头+列表+操作）/ I（环形指标行+统计）
├── 4×4：模板 J（Grid 信息组合）/ K（多设备网格面板）
└── 尺寸不确定 → 优先选满足硬限的最小尺寸；若估算超过硬限则升级

选定模板后：
1. 阅读本文件对应尺寸的布局铁律和反模式
2. 若用户未指定尺寸，内部记录判断依据：信息槽位数、按钮数、可见文本行数、是否需要 List/Grid
3. 若选 2×2，先匹配新模板抽象中的三类稳定骨架；使用渐变时必须从语义色板速查表选择色彩，使用图片时必须遵守图像背景叠层规则
4. 根据需求调整信息区内容
5. 进入 guide.md 执行 DSL 生成
```

**2×2 新模板参考实现：** 见 [dsl-templates/](dsl-templates/)：

- [top-left-weather-schedule.dsl.jsonl](dsl-templates/top-left-weather-schedule.dsl.jsonl)：分层主任务卡，适合主指标 + 上下文 + 底部 CTA。
- [parents-care-weather.dsl.jsonl](dsl-templates/parents-care-weather.dsl.jsonl)：上下分区行动卡，适合顶部大状态 + 底部详情卡/圆形操作。
- [free-clip-2-widget/free-clip-2.dsl.jsonl](dsl-templates/free-clip-2-widget/free-clip-2.dsl.jsonl)：图像背景叠层卡，适合产品/设备状态。
