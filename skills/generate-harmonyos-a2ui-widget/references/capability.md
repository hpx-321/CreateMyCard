# 卡片场景能力范围

> **用途：** 生成 DSL 前判断用户需求是否在当前渲染器能力范围内。无法满足时明确告知用户并给出替代方案；可满足后再进入 [template.md](template.md)。

## 当前能力边界

### 支持场景

按任务形态而不是业务名判断能力。天气、日程、健康、股票、快递、设备、内容摘要等都先映射到下表的信息角色。

| 任务形态 | 结构信号 | 推荐组件 |
|------|---------|---------|
| **单主指标/结论** | 1 个数值、状态、倒计时、温度、余额、等级、告警结论 | `Text` + `Column`/`Row` |
| **主信息 + 上下文** | 主指标旁有时间、地点、来源、趋势、状态解释 | `Text` + 半透明 `Column`/`Row` |
| **轻操作入口** | 1-3 个明确动作，如打开、拨打、控制、查看、导航 | `Button` 或可点击 `Row` + `Text` |
| **短列表/明细** | 2-5 条同类项，如待办、日程、消息、物流节点、设备状态 | `List` + 模板子节点，或固定 `Row`/`Column` |
| **指标仪表/进度** | 完成度、比例、环形/条形状态、用量 | `Progress` + `Text` |
| **条件状态** | 加载、空态、异常、断点布局、不同状态内容 | `If` + `$__WindowWidthBreakpoint`/DataModel |
| **组合面板** | 多个同级模块、二维网格、统计面板、设备矩阵 | `Grid`/`Row`/`Column` 组合 |

### 支持尺寸

| 卡片规格 | 典型尺寸 (vp) | 适用场景 |
|---------|-------------|---------|
| 1×2 | ~80×160 | 单指标微卡、图标+数值 |
| 2×2 | ~160×160 | 主指标+上下文、状态+操作、短列表 |
| 2×4 | ~160×320 | 纵向详情、列表卡、多条明细 |
| 4×4 | ~320×320 | 仪表盘、大图+列表、组合信息卡 |

### 支持能力

- **v0.9 协议** 全部消息类型：`createSurface`、`updateComponents`、`updateDataModel`、`deleteSurface`
- **鸿蒙扩展组件** 当前 Docs 聚合 Schema 中的简名组件；旧式 `Extended.*` 仅作为兼容输入处理
- **DataModel 路径绑定**（JSON Pointer）与 `{{ }}` 表达式
- **动态列表**（模板子节点 + `$item`/`$index`）
- **响应式自适应**（`$__WindowWidthBreakpoint` + `If` 组件 + 三元表达式）
- **深浅色模式**（`$__ColorMode`）
- **事件处理链**（`onClick`、`onAppear`、`onChange` 等 EventHandler 数组）
- **15 种通用样式**（背景、边框、阴影、渐变、显隐、权重等）

## 不支持场景

以下需求超出当前渲染器或 Skill 范围，**必须明确告知用户**：

| 不支持的场景 | 原因 | 替代方案 |
|-------------|------|---------|
| **标准组件目录卡片** | 本 Skill 仅针对扩展组件目录 | 使用标准 Catalog 需走 A2UI Basic Catalog 协议 |
| **表单输入型卡片** | 桌面卡片应以展示+轻交互为主 | 输入类控件（`TextInput`、`Select`）仅在需求明确且有端侧闭环时使用 |
| **视频/音频播放** | `Web` 不适合桌面卡片，`Video`/`AudioPlayer` 为标准组件 | 建议在完整页面中使用 |
| **多页导航** | `Navigation`/`NavContainer` 适用于完整页面，桌面卡片应单 Surface | 使用单卡片 + 条件切换 (`If`) |
| **Tab 切换卡片** | `Tabs` 在窄小卡片中体验不佳 | 拆分为多张独立卡片 |
| **地图/定位** | 无对应组件 | 使用文本展示位置摘要 |
| **实时流数据** | 刷新策略由端侧运行时负责 | 在 DataModel 中保留更新路径，端侧定时推送 |
| **复杂动画/转场** | 协议未定义动画语义 | 使用 `onAppear` 事件配合端侧动画 |
| **服务端推送** | 推送通道超出 DSL 范围 | 端侧建立长连接后调用 `updateDataModel` |
| **用户认证/权限** | 卡片不承载登录态 | 权限由宿主应用统一管理 |
| **混用两套 Catalog** | 同一 Surface 不能同时使用标准+扩展组件 | 选择其一；桌面卡片推荐扩展 Catalog |

## 判断流程

收到用户需求后按以下顺序判断：

```
1. 拆解需求 → 抽取信息角色：primary、context、details、action、state
2. 匹配能力 → 是否可由支持的任务形态组合表达？
3. 检查尺寸 → 用户是否指定尺寸？内容量是否匹配所选尺寸？
4. 检查交互 → 是否只是轻操作？事件是否有明确契约？是否误用了输入控件？
5. 结论：
   ├── 全部匹配 → 进入模板选择（见 template.md）
   ├── 部分不匹配 → 告知受限部分 + 给出替代方案
   └── 核心不匹配 → 明确告知无法满足 + 说明原因 + 建议替代
```

## 告知用户的规范

当需求无法满足时，按以下格式告知：

```
⚠️ 当前渲染器不支持 [具体需求]，原因是 [技术原因]。

替代方案：[可行的替代实现方式]

如果接受替代方案，我将继续生成 DSL。
```
